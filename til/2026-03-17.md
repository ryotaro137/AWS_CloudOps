## AWS CDK
- **概要**: 汎用プログラミング言語（TypeScript, Pythonなど）を用いてクラウドインフラストラクチャをコードとして定義・構築できるAWS提供のフレームワーク。内部的にCloudFormationテンプレートを生成・デプロイする。
- **試験ポイント**: CloudFormationとの違い（プログラミング言語が使えるか）、CDKの主要コンポーネント構成（App, Stack, Construct）、`cdk synth`コマンドによるCFnテンプレート生成、`cdk deploy`コマンドの役割などが頻出。
- **実務での使い方**: アプリケーションのコードと同じ言語・リポジトリでインフラも管理できるため、モダンなWeb・サーバーレス開発（Lambda + API Gatewayなど）で非常に重宝される。よく使うリソース群をパッケージ（Construct）化して社内で再利用する構成がベストプラクティス。
- **副業での需要**: ★★★★★（最高）。IaC化の需要は圧倒的で、「手動で作った環境をCDKでコード化してほしい」「既存のシステムを複製しやすくCDK化したい」という案件が多数。特にTypeScriptでの開発スキルと組み合わせると、50〜150万円/案件レベルの高単価案件の獲得に直結する。

- NATはTCP、UDP、ICMPのみ対応している。

## CloudFrontとALB（リスナー/スティッキーセッション）の連携
- **概要**: CloudFrontの背後にApplication Load Balancer (ALB) を配置する構成において、ALBの「リスナールール（パスベースルーティング等）」や「スティッキーセッション（セッション維持機能）」を正しく動作させるためのCloudFront側の設定。
- **試験ポイント**: ALB側でスティッキーセッションを有効にしていても、**CloudFrontの「キャッシュビヘイビア（Cache Behavior）」設定で、ALBが発行するSession Cookie（例: `AWSALB`）をオリジンにフォワード（転送）するように設定**しなければ、セッション維持が効かず毎回異なるターゲットに振り分けられてしまうというトラブルシューティング問題が頻出。
- **実務での使い方**: ステートフルな（セッション情報をEC2内に持つ）レガシーアプリの前にCDNを配置する場合に必須の設計。静的コンテンツ（`/images/*` など）はCloudFrontでキャッシュさせ、動的リクエスト（`/api/*` など）はキャッシュせずALBへCookieごと転送し、リスナールールでターゲットグループに振り分ける構成がベストプラクティス。
- **副業での需要**: ★★★★（高い）。「既存の重いサイトの前にCloudFrontを置いて高速化・負荷分散したい」というWebパフォーマンス改善案件で頻繁に登場。「静的/動的パスの切り分け設計」とセットで、10〜30万円/案件の改善提案や構築としてよく出回る。
