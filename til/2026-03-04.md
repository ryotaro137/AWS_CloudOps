## AWS Network Firewall
- **概要**: VPCレベルで動作するマネージド型ファイアウォール。ステートフルなパケット検査とディープパケットインスペクション（DPI）により、トラフィックの高度な制御を実現する。
- **試験ポイント**: 
    - **ステートフルな検査**: セキュリティグループ（L4）では不可能な、L7レベルのペイロード検査やドメインベースのフィルタリングが可能。
    - **AWS Managed Rules**: 既知の脆弱性やボットネット等に対するシグネチャがAWSにより提供・更新される。
    - **WAFとの棲み分け**: WAFはアプリケーション（HTTP/S）保護、Network Firewallはネットワークレイヤー全体の保護。
- **実務での使い方**: 
    - **集中型検査構成**: Transit Gatewayを介して、全VPCのインバウンド・アウトバウンド通信を「インスペクションVPC」に集約して一括検査する構成が鉄板。
    - **ホワイトリスト管理**: セキュリティグループでは難しい「FQDN（完全修飾ドメイン名）ベース」のアウトバウンド通信制限に非常に有効。
- **副業での需要**: ★★★★ (セキュリティ・運用)。大規模環境のセキュリティ統制強化や、従来のプロキシサーバー等の運用コスト削減を目的とした移行案件での提案が強力。設計・要件定義から入ることで高単価（月額30〜60万円〜）の支援に繋がる。

## Auto Scaling とマルチリージョン高可用性設計
- **概要**: Auto Scaling グループ (ASG) は単一リージョン内でのみ動作。マルチリージョンで高可用性を実現するには、各リージョンに ASG を作成し Route 53 等でトラフィックを振り分ける構成が必要。
- **試験ポイント**: 
    - **リージョン境界**: ASG は「単一リージョンの複数 AZ」を跨ぐことはできるが、別リージョンには及ばない。
    - **Route 53 との組み合わせ**: フェイルオーバー（Active-Standby）やレイテンシールーティング（Active-Active）など、Route 53 のポリシーとのセットで問われる。
    - **コストと遅延**: リージョン間のデータ転送コストやネットワーク遅延も、アーキテクチャ選定の重要な判断軸。
- **実務での使い方**: 
    - **DR（災害復旧）対策**: 東京リージョンが全停止した場合に備え、大阪リージョンに ASG を待機、または少量起動（Pilot Light/Warm Standby）させておく設計。
    - **グローバル展開**: ユーザーに近いリージョンで処理を行うため、各大陸のリージョンに ASG を配置し Route 53 Geolocation/Latency routing を活用する。
- **副業での需要**: ★★★★★ (アーキテクチャ設計・コスト最適化)。「単純にマルチリージョンにすれば良い」というわけではなく、要件（RTO/RPO）とコスト・複雑性のバランスを考えた「適正な設計」を提案できるスキルは、上流工程の案件（月単価80万円〜）で非常に重宝される。

## CloudWatch Agent: procstat プラグインによるプロセス監視
- **概要**: EC2インスタンス内の「個別プロセス（Apache, Nginx, DB等）」のCPU、メモリ、PID情報を収集・送信できる。
- **試験ポイント**: 
    - **インスタンス内メトリクス**: CloudWatch 標準メトリクス（CPU, ネットワーク, 基本ディスク）では見えない「特定のプロセスの挙動」を把握するための最適解。
    - **アラーム統合**: 個別プロセスのリソース超過やプロセスダウンを検知し、自動復旧アクションに繋げる設計。
- **実務での使い方**: 
    - **「特定のプロセスが重い」問題の特定**: 全体のCPU使用率が高騰した際、どのアプリケーションが原因かを即座に特定するために必須のプラグイン。
    - **監視・運用保守の自動化**: 一度設定すれば、SSM等を用いて全インスタンスに横展開が可能。
- **副業での需要**: ★★★★★ (監視・アラート設計)。「サーバー全体ではなく、特定のアプリケーションの挙動を監視したい」という現場の切実な課題を、低コスト（Managedサービスの範疇）で解決できる。既存のZabbix等からのAWSネイティブ監視への移行（20〜50万円〜/月）などで大きな成果を出せる。

## ElastiCache for Redis: クラスターモード無停止リシャーディング
- **概要**: クラスターモード有効なRedisで、サービスを停止することなくシャードの追加・削除（オンラインリシャーディング）が可能。
- **試験ポイント**: 
    - **可用性**: データの再配布中も読み書きを継続できる「無停止」の特性が、ダウンタイムを許容できないシナリオでの正解となる。
    - **スケールアウト**: 書き込み負荷（CPU 100%）が高まった際の究極の対策（単なるリードレプリカ追加では解決しない場合）。
- **実務での使い方**: 
    - **スパイク対応**: オークションやセールなどの「イベント時の負荷変動」に対し、事前に、あるいは発生時に動的にノードを追加してトラフィックを分散させる。
- **副業での需要**: ★★★★★ (アーキテクチャ設計・コスト最適化)。「アクセス急増でDBが詰まってサイトが落ちる」というクリティカルな課題を、最小限のダウンタイム（というかゼロ）で解決する設計を提案できる。サイトパフォーマンスの改善コンサル（月額40万円〜）などで非常に説得力のある武器になる。

## AWS Secrets Manager: データベース認証情報の自動ローテーション
- **概要**: RDSやAuroraなどの認証情報を安全に管理し、Lambdaを介してパスワードを自動更新（ローテーション）するマネージドサービス。
- **試験ポイント**: 
    - **自動ローテーション**: RDS/Aurora/Redshift等に対して「組み込みのローテーションロジック」が提供されており、ユーザーがコードを書かずに自動更新を実現できる。
    - **セキュリティ**: KMSによる暗号化、IAMによる詳細なアクセス制御、CloudTrailによる監査ログが完備されている点。
- **実務での使い方**: 
    - **アプリケーションの透過的アクセス**: アプリ側でSecrets Manager APIを叩くように実装することで、パスワード変更時もコード修正や再デプロイなしで最新の認証情報を利用可能にする。
- **副業での需要**: ★★★★★ (セキュリティ運用・ガバナンス強化)。「パスワードが環境変数にべた書き」「数年更新されていない」という脆弱な現場に対し、工数をかけずに「最高水準の認証情報管理」を導入できる提案は、セキュリティ改善案件（30〜60万円〜）で非常に高く評価される。

## VPC Flow Logs による ECS タスク間通信の可観測性
- **概要**: VPC内のENIを出入りするトラフィック情報をキャプチャし、ECSタスクレベルでの詳細なネットワークフロー（接続先、ポート、データ量）を可視化する。
- **試験ポイント**: 
    - **ENI単位のキャプチャ**: ECS Fargate等の各タスクに割り当てられたENIで有効化することで、マイクロサービス間の複雑な通信関係を記録できる点。
    - **アクション（ACCEPT/REJECT）**: セキュリティグループ等で「拒否された通信（REJECT）」を特定し、セキュリティ分析や接続不良のトラブルシューティングに活用する。
- **実務での使い方**: 
    - **トラフィック分析**: CloudWatch Logs Insightsを用いて「どのタスクからどのDBへ、どれだけの通信が発生しているか」をクエリし、設計の妥当性検証や異常検知に役立てる。
- **副業での需要**: ★★★★ (監視・トラブルシューティング)。「原因不明の通信エラー」や「セキュリティ監査のための通信ログ保存」という依頼に対し、低オーバーヘッド（性能劣化なし）で確実な証跡を残せるソリューションとして、保守・調査案件（スポット5〜15万円〜）で重宝される。

## ECS ネットワークモードの比較と選定
- **概要**: ECSでは `awsvpc`, `bridge`, `host` の3つの主要なネットワークモードがあり、それぞれ分離レベルやパフォーマンスが異なる。
- **試験ポイント**: 
    - **awsvpc**: Fargateでは必須。タスクごとに専用ENI/IPが付与され、タスク単位でセキュリティグループを適用できる「最もセキュア」なモードとして選ばれる。
    - **bridge**: Dockerデフォルト。ホストのIPを共有し、ポートマッピングを使用する。タスク間のネットワーク分離が不十分という点が弱点。
    - **host**: コンテナがホストのネットワークスタックを直接利用。オーバーヘッドが最小で低レイテンシーだが、ポートの重複（同一ホストで同ポートの複数タスク起動不可）が制約。
- **実務での使い方**: 
    - **標準構成**: 特段の理由がない限り、高いセキュリティと可観測性を得られる `awsvpc` モードを選択するのがベストプラクティス。
    - **性能特化型**: 音声配信やリアルタイム通信など、極限までレイテンシーを削る必要がある場合にのみ `host` モードを検討する。
- **副業での需要**: ★★★★★ (アーキテクチャ設計)。「なんとなく動けば良い」から脱却し、セキュリティ要件や監視要件に合わせて最適なモードを選択できるスキルは、モダン開発チームの支援（月額40〜80万円〜）においてプロフェッショナルとして信頼を得るために不可欠。

## Amazon EKS: クラスター接続と主要コンポーネント
- **概要**: `kubectl` によるクラスター操作の基本、`kube-proxy` や `Fargateプロファイル` などの構成要素、および `EKS Connector` による外部クラスター統合。
- **試験ポイント**: 
    - **接続認証**: `aws eks update-kubeconfig` による `kubeconfig` 生成と、IAM認証・RBAC統合の仕組み。
    - **kube-proxy**: 各ノードで動作し、Serviceリソース（iptables等）を通じたルーティングとロードバランシングを担当する。
    - **Fargateプロファイル**: 名前空間やラベルでポッドを指定し、サーバーレス実行環境（Fargate）に振り分ける設定。
    - **EKS Connector**: オンプレミスや他クラウドの K8s クラスターを AWS コンソールで一元管理するための機能。(`eks-connector.yaml` を使用)
- **実務での使い方**: 
    - **運用自動化**: CI/CD パイプライン内で `update-kubeconfig` を実行し、安全にデプロイメントを行う構成。
    - **サーバーレス運用**: インフラ管理を最小化するため、ステートレスなワークロードを Fargate プロファイルで管理する。
- **副業での需要**: ★★★★★ (EKS/Kubernetes構築・運用)。EKSの導入・移行・運用支援は、非常に単価が高い（月額80〜150万円〜）最先端領域。認証・認可の正しい理解や, Fargate活用による運用負荷軽減の提案ができるエンジニアは圧倒的に不足しており、非常に高単価な案件を狙いやすい。

## ディザスタリカバリ (DR) と Aurora Global Database
- **概要**: RPO/RTOに基づいたDR設計。Aurora Global Databaseを活用することで、リージョン全体の障害に備えたクロスリージョンDRを実現する。
- **試験ポイント**: 
    - **RPO (Recovery Point Objective)**: 許容できるデータ損失時間。RPO 20秒なら、最大20秒分の損失を許容することを意味する。
    - **Aurora Global Database**: 物理レプリケーションにより1秒未満のラグを実現。最大5つのセカンダリリージョンをサポートし、1分未満でのフェイルオーバーが可能。
    - **Multi-AZとの違い**: Multi-AZは「リージョン内のAZ障害」に対する高可用性(HA)。リージョン全体の障害には「クロスリージョン(Global DB)」が必要。
    - **レプリカラグ**: システムの負荷やNW特性で決まる「結果（メトリクス）」であり、パラメータとして直接設定するものではない。
- **実務での使い方**: 
    - **金融・ミッションクリティカル系**: 災害対策（DR）として、東京から大阪へ Global Database を構成。昇格（Promotion）手順を自動化・文書化しておく。
- **副業での需要**: ★★★★★ (BCP策定・アーキテクチャ設計)。「リージョンが丸ごと落ちた時にどうするか」という経営レベルの懸念（BCP対策）に対し、具体的な技術選定と復旧手順を提示できるエンジニアは極めて稀。高単価のコンサル案件や技術顧問（月額50〜100万円〜）へのゲートウェイとなる。
