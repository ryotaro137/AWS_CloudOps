## aws sts と aws configure によるローカル環境構築
- **概要**: `aws configure`コマンドを用いてAWSアクセスキーやデフォルトリージョンを設定し、`aws sts get-caller-identity`コマンドなどを活用して自分の認証情報が正しく設定・認識されているかを確認する方法。これにより、ローカルターミナルからAWSリソースをAWS CLI経由で安全に操作するための基盤が整う。
- **試験ポイント**: 
    - **認証情報の優先順位**: `aws configure`で設定した情報は`~/.aws/credentials`などに保存されるが、試験では「環境変数、CLI設定ファイル、IAMロール（EC2/ECS）」のうちどれが優先されるかという順番が頻出。
    - **STSの役割**: AWS Security Token Service (STS) を利用した一時的認証情報の発行（`AssumeRole`によるクロスアカウントアクセスなど）の仕組みが問われる。
- **実務での使い方**: 
    - **プロファイルの使い分け**: 現場では開発・検証・本番環境など複数アカウントを扱うため、`aws configure --profile <環境名>`でプロファイルを分割し、操作ミスを防ぐ構成にするのが基本。
    - **よりモダンな認証**: ローカルPCに永続的なアクセスキー（長寿命クレデンシャル）を置くのは漏洩リスクがあるため、実務ではAWS IAM Identity Center (旧AWS SSO) と連携した一時クレデンシャル取得（`aws sso login`）に移行しつつある点にも留意する。
- **副業での需要**: ★★★★★ (IaC / 環境構築)。どのようなAWS案件（Terraform構築、Lambda開発、トラブル調査など）を受けるにしても、自分の環境からセキュアに客先のAWS環境へアクセスするための第一歩として必須のスキル。「複数アカウント間のスイッチロールを理解しており、安全なアクセス管理ができる」ことはプロとしての最低条件になる。
