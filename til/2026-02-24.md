## 学習内容

## CloudFormation ドリフト検出 vs CloudTrail
- **概要**: ドリフト検出は「設定のズレ（結果）」を、CloudTrailは「操作の履歴（過程）」を特定する機能。
- **試験ポイント**: ドリフトはIaCの整合性、CloudTrailはユーザーの特定や監査。どちらで「何」が分かるかの違いが頻出。
- **実務での使い方**: 運用中の不用意な手動変更をドリフトで検知し、CloudTrailで「誰がやったか」を裏取りして、運用ルールの改善に繋げる。
- **副業での需要**: ★★★★★。IaC移行案件において、手動操作を禁止するためのガバナンス設計として提案のセットにするのが鉄板。

## RDS Proxy vs リードレプリカ
- **概要**: RDS Proxy は「コネクション（接続数）管理」、リードレプリカは「読み取り負荷（クエリ数）分散」のためのサービス。
- **試験ポイント**: Lambda との連携なら Proxy、読み取りパフォーマンス向上ならレプリカ。解決したいボトルネックが「接続数」か「データ処理量」かを見極める。
- **実務での使い方**: サーバーレス構成では Proxy を必須級で検討し、読み取りリクエストが支配的なシステムではレプリカでスケールアウトを設計する。
- **副業での需要**: ★★★★。どちらも「既存システムの高負荷対策」として非常に引き合いが多い知識。

## AWS Organizations
- **概要**: 複数のAWSアカウントを統合管理し、一括請求や一元的な権覇制御（SCP）を行うサービス。
- **試験ポイント**: SCPによるガードレール、一括請求によるコスト削減、新規アカウントへの自動デプロイ（StackSets連携）が頻出。
- **実務での使い方**: 爆発半径の抑制（環境分離）のために必須。Control Tower と組み合わせてガバナンスの効いたアカウント量産体制を作る。
- **副業での需要**: ★★★★★。スタートアップの初期構築や、既存環境のセキュリティ見直し（アカウント整理）案件で極めて重宝されるスキル。

## Lambda の実行モデルと同時実行制御
- **概要**: デフォルトの「オンデマンド」に加え、上限保証・制限の「予約済みの同時実行」、Java高速化の「SnapStart」がある。
- **試験ポイント**: コールドスタート許容ならオンデマンド。特定関数の暴走阻止やバックエンド保護なら「予約済みの同時実行」。Javaの遅延対策なら「SnapStart」。
- **実務での使い方**: 読み込みリクエストが爆発するAPIではDB保護のために「予約済みの同時実行」でキャップをかけ、非同期処理のDead Letter Queue (DLQ) でエラー時の再試行を担保する。
- **副業での需要**: ★★★★。サーバーレスのパフォーマンス改善案件で、コスト（Provisioned）と性能（On-demand/SnapStart）のバランスを提案できるスキルは高単価。

## CloudFormation テンプレートの主要セクション
- **概要**: Resources（必須）の他に、動的制御を行う Parameters, Mappings, Conditions, Outputs などで構成される。
- **試験ポイント**: 再利用性の Parameters、静的変換の Mappings、環境出し分けの Conditions、スタック間連携の Outputs の役割の違いを理解する。
- **実務での使い方**: Metadata でコンパネの入力項目を整理し、Mappings でAMI等の管理を自動化することで、誰が実行しても同じ環境ができる「冪等性（べきとうせい）」と「操作性」を確保する。
- **副業での需要**: ★★★★★。ハードコードされたレガシーなテンプレートの「汎用化・モジュール化」は需要が非常に高く、保守性の高いコードを書けるエンジニアは重宝される。

## KMS キーポリシー
- **概要**: KMSキー自体に付着するリソースベースのポリシー。KMSへのアクセス制御の主導権を握る。
- **試験ポイント**: IAMポリシーだけでは不十分。キーポリシー側で「IAMによる権限付与を許可する」設定がないとIAMの設定が効かない。クロスアカウントアクセス時の必須設定。
- **実務での使い方**: ルートユーザーへの全権委譲設定を忘れずに入れ、鍵の「管理者」と「使用者」の権限を分離する。また、外部サービス（CloudWatch Logs等）が鍵を使うための許可を個別に定義する。
- **副業での需要**: ★★★★。暗号化要件がある案件でのトラブル（権限不足でリソースが作れない等）の多くはキーポリシーに関係するため、ここを即座にトラブルシュートできると頼りにされる。

## Aurora S3 エクステンション (aws_s3)
- **概要**: Auroraから直接S3のデータをインポート・エクスポートできる機能。
- **試験ポイント**: SQL（LOAD DATA 等）のみでS3連携が可能な点、中間サーバーが不要になる点、AuroraへのIAMロール付与が必要な点が頻出。
- **実務での使い方**: 大量データのバルクロードや、分析用データのS3への書き出しに活用。アプリケーションコードを書かずにインフラ（DB）の機能だけでデータ連携を完結させる。
- **副業での需要**: ★★★★。データ移行やバッチ処理の高速化案件で、開発工数を抑えつつ高性能な仕組みを提案する際に非常に有効。

## OpenID Connect (OIDC)
- **概要**: 外部の認証サーバーを利用して本人確認を行い、一時的なAWSアクセス権限を取得するプロトコル。
- **試験ポイント**: 長期的な「アクセスキー」の排除、IAM IDプロバイダーの設定、一時的なクレデンシャルの発行がキーワード。
- **実務での使い方**: GitHub Actions からのデプロイ時に必須。秘密情報をGitHub側に持たせず、ジョブ実行時のみ必要なロールを引き受けさせる（フェデレーション）。
- **副業での需要**: ★★★★★。セキュリティ意識の高いモダンな開発現場では必須スキル。アクセスキー運用からの脱却を提案できると単価と信頼が大きく向上する。

## S3 イベント通知のプレフィックスフィルター
- **概要**: オブジェクト名の先頭（パス）に基づいてイベント通知の対象を絞り込む機能。
- **試験ポイント**: 不要な後続処理（Lambda等）の起動を抑える手法。同一バケット内でプレフィックスが重複（包含関係含む）する設定はできない制約が頻出。
- **実務での使い方**: `uploads/` フォルダのみを対象にして無限ループを防止したり、拡張子フィルター（サフィックス）と組み合わせて特定ファイルのみを処理するイベント駆動設計に活用。
- **副業での需要**: ★★★★★。S3を起点とした自動化処理は案件数が非常に多く、フィルターを適切に使い分けられることは必須スキル。

## EventBridge Pipes / SSM Automation / ステップスケーリング
- **概要**: Pipesはサービス間の低コストな橋渡し、Automationは運用手順の自動化、ステップスケーリングは段階的なリソース増減。
- **試験ポイント**: Pipesは「Lambda不要の連携」、Automationは「Config連携による自動是正」、ステップスケーリングは「超過量に応じた段階的調整」。
- **実務での使い方**: Pipesでサーバーレス構成をシンプルに保ち、SSM Automationで手動運用のミスを排除、ステップスケーリングでスパイクに対応する。
- **副業での需要**: ★★★★★。特に運用自動化（SSM）と連携効率化（Pipes）は、技術選定のセンスをアピールできる高単価領域。
