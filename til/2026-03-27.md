## 学習内容

## AMI (Amazon Machine Image)
- **概要**: EC2インスタンスを起動するためのテンプレート。ルートボリューム、追加EBS、メタデータ、スナップショット等を含む。
- **試験ポイント**: AMIの構成要素（EBSスナップショットが含まれる点）や、別リージョン/アカウントへのコピー手順が問われやすい。
- **実務での使い方**: ゴールデンイメージ（OS設定やミドルウェア導入済みのAMI）を作成し、Auto Scalingグループの起動テンプレートとして利用する。
- **副業での需要**: ★★★☆☆ (AMIの定期バックアップ自動化やゴールデンイメージ作成パイプライン構築の案件がある)

## ファイルシステム整合性とCreateImageのrebootパラメーター
- **概要**: 整合性とはディスクデータの一貫性のこと。AMI作成時（CreateImage）の`reboot=true`（デフォルト）は再起動で未書き込みデータをディスクに反映し整合性を保つ。
- **試験ポイント**: `NoReboot`パラメーターの意味。`true`ならダウンタイムなしだが整合性リスクあり。DB等の整合性が重要な場合は再起動（または静止点確保）が必要と問われる。
- **実務での使い方**: 本番DBサーバーのバックアップでは、ダウンタイムを避けるためにリードレプリカから取得するか、アプリケーションレイヤーで静止点を設けてバックアップを行う。
- **副業での需要**: ★★★★☆ (無停止バックアップ要件や、ダウンタイム調整を含む確実なバックアップ設計・運用は頻出)

## Amazon EventBridgeとLambdaによる自動化
- **概要**: EventBridgeのCron/レート式でLambdaを定期実行（例: boto3で`create_image` API呼び出し）するサーバーレスな自動化手法。
- **試験ポイント**: CloudWatch Events（現EventBridge）を用いた定期バックアップ構成や、エラーハンドリング（通知や再試行）の仕組みが問われる。
- **実務での使い方**: インスタンスに特定のタグ（例：`Backup: Daily`）を付け、Lambda側でそのタグを持つ全インスタンスのAMIを日次取得するよう構成する。
- **副業での需要**: ★★★★★ (定期バックアップ自動化スクリプト自体の作成や環境改善案件は小規模ながら非常に多く、IaCの第一歩としても需要が高い)

## AWS BackupとAMI作成の違い
- **概要**: AWS BackupはEBS/RDS/DynamoDB等の一元的なバックアップサービス。AMI単体の作成と異なり、整合性制御の自由度は異なるが長期保存・一元管理に強い。
- **試験ポイント**: リソースの一元的なバックアップ管理やコンプライアンス要件（クロスアカウント・別リージョンへのコピー、Vaultロック等）が求められる場合にAWS Backupが正解になる。
- **実務での使い方**: 個別のLambda等でのスクリプト管理を廃止し、AWS Backupのバックアッププランを利用して全体のバックアップ運用を標準化する。運用負荷を大きく下げる。
- **副業での需要**: ★★★★☆ (既存の個別バックアップスクリプト群をAWS Backupへモダン化・一元化・IaC化する移行案件が増加中)

## AWS Network Firewallの動作スコープ
- **概要**: AWS Network Firewallはサブネット単位ではなくVPCレベルで動作・保護を提供するマネージドファイアウォール。エンドポイント自体は特定AZのサブネットに配置される。
- **試験ポイント**: 「どのレベルで動作するか」の引っかけ問題に注意。配置先はサブネットだが、保護範囲やルーティング制御の適用スコープは「VPC全体」である点がよく問われる。
- **実務での使い方**: Transit Gatewayと組み合わせて、複数VPC間のトラフィックやオンプレミス・インターネットゲートウェイ宛の通信を「検査用VPC (Inspection VPC)」に集約してNetwork Firewallで一元管理する構成が定番。
- **副業での需要**: ★★★★☆ (セキュアなマルチアカウント構成・Transit GatewayやControl Tower導入を伴う大規模ネットワーク設計案件で頻出)

## AWS IAM Identity CenterとLDAP連携の制限
- **概要**: 複数AWSアカウントとアプリへのSSOを提供するIAM Identity Center（旧AWS SSO）は、オンプレミスLDAPディレクトリとの直接同期・レプリケーション機能をサポートしていない。
- **試験ポイント**: 「既存のLDAPをIAM Identity Centerに直接同期する」という選択肢は誤り。SAMLベースのフェデレーション、AD Connector経由のActive Directory連携、またはSCIMプロトコル統合が正解となる。
- **実務での使い方**: オンプレミス環境にActive Directoryがある場合は、AD Connectorを介して連携するか、Azure AD (Entra ID) や Okta 等の外部IdPとSAML/SCIM連携する構成が一般的。
- **副業での需要**: ★★★★★ (マルチアカウント管理におけるSSO導入や、既存IdP(Google WorkspaceやAzure ADなど)との統合案件は非常に多い)

## Aurora Global Databaseとその他の高可用性構成
- **概要**: Aurora Global Databaseは複数リージョン間の物理レプリケーション（ラグ通常1秒未満）を提供し、真のクロスリージョンDR（ディザスタリカバリ）を実現する。最大5リージョンをサポート。
- **試験ポイント**: リージョン障害の対策としては「Aurora Global Database」一択。Multi-AZやAurora Replicaは単一リージョン内の障害（AZ障害）への対応用であり、リージョン全体のDR機能ではない点が出題の中心。
- **実務での使い方**: RTO（目標復旧時間）/RPO（目標復旧時点）が極めて厳しい金融・グローバルサービスで導入する。Aurora Serverless v2と組み合わせることで容量の自動スケーリング要件にも対応。
- **副業での需要**: ★★★☆☆ (Global Database自体の設計はエンタープライズ領域が中心。副業の規模感では通常のMulti-AZ構築やAurora Serverless v2でのスケーリング最適化案件の方が需要がある)

## AWS KMSとVPCエンドポイント (PrivateLink)
- **概要**: AWS KMSはVPCインターフェースエンドポイント（AWS PrivateLink）をサポート。これにより、インターネットを経由せずVPC内からセキュアにKMS APIへアクセス可能。
- **試験ポイント**: VPCエンドポイントのタイプ（Gateway型はS3/DynamoDBのみ、KMS等はInterface型）の引っかけや、閉域網からKMSにアクセスする要件で「Interfaceエンドポイントの作成」が答えになる問題が頻出。
- **実務での使い方**: セキュリティ要件の厳しいシステムにおいて、プライベートサブネット内のリソースがデータの暗号化・復号のためにKMSを呼び出す際に必ず構成する。プライベートDNSを有効化して透過的にアクセスさせるのがセオリー。
- **副業での需要**: ★★★★★ (「完全閉域網でのシステム構築」はよくある要件であり、KMSやSSM向けのVPCエンドポイント設計・IaC実装はインフラ案件で必須スキル)
