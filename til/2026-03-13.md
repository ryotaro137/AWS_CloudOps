# 2026-03-13 の学習記録

## AWS CDKにおける `cdk.out/manifest.json` の役割

一言で言うと、**`manifest.json`は、CDKアプリをデプロイするための「設計図の目次・指示書」**です。

CDKでコードを書いて `cdk synth` を実行すると、そのプログラムが実行され、最終的に `cdk.out` フォルダにいろいろなファイルが出力されます。この出力された一連のファイル群のことを「**Cloud Assembly (クラウド・アセンブリ)**」と呼びます。

`manifest.json` は、この Cloud Assembly の全体像を記述した最も重要なファイルであり、以下のような不可欠な役割を担っています。

### 1. 「何を・どこに」デプロイするかを定義する (Artifactsの管理)
`cdk.out` フォルダの中には、通常 `.template.json` のような個別のCloudFormationテンプレートファイルや、Asset（LambdaのソースコードやDockerイメージなど）が生成されます。

`manifest.json` は、「このアプリにはどんな成果物（Artifact）が含まれているのか」をリストアップしています。
たとえば、「スタックのテンプレートはどこにある」「デプロイ先の環境はどこか」といった情報がすべてここに書かれています。

### 2. デプロイの「依存関係と順番」を管理する
複数のStackがある場合、依存関係が発生します。
`manifest.json` には、各成果物間の依存関係が明確に記録されます。これにより、CDK CLI (`cdk deploy`) はどの順番でAWSにリソースを作成していけばいいかを正確に判断できます。

### 3. プログラミング言語とデプロイエンジンの「分離（デカップリング）」
これが最も重要な必要性です。

AWS CDKを使えば、Python、TypeScriptなど好きな言語でインフラを定義できます。ですが、実際のAWSの裏側（CloudFormationやデプロイツール等）は、それらの言語を直接理解できるわけではありません。

そこで、次のようなプロセスを踏みます：
1. **Synth (合成) フェーズ**: あなたが書いたPython/TSコードが実行され、言語に依存しない普遍的な形式（Cloud Assembly = `manifest.json` + テンプレート群）が生成される。
2. **Deploy (展開) フェーズ**: デプロイツール (`cdk deploy` コマンドなど) は、**プログラムコードは一切見ず、この `manifest.json` だけを読み取って**デプロイ作業を行う。

つまり、`manifest.json` がないことには、CDK CLIはデプロイが不可能になります。
手動・CI/CD問わず必須となる、機械向けのファイルです。

## `cdk bootstrap` と `cdk deploy` の違い

AWS CDKにおいて、`cdk bootstrap` と `cdk deploy` はどちらも重要なコマンドですが、役割と実行するタイミングが明確に異なります。

### `cdk bootstrap`：土台・準備（環境構築）
*   **役割**: CDKがデプロイ作業を行うために必要な「AWS側の土台（基礎環境）」を作ること。S3バケットやECRリポジトリ、IAMロールなどを作成します。
*   **実行タイミング**: 特定のAWS環境（アカウント + リージョン）を使用する一番最初だけ実行します。

### `cdk deploy`：本体の構築（実際のアプリの展開）
*   **役割**: スクリプトのコードに基づいて、AWS上に実際のインフラリソース（VPC、EC2など）を作成・更新すること。
*   **実行タイミング**: インフラのコードを実際にAWSに反映・更新させたい時に毎回実行します。

## `build` と `cdk bootstrap` の関係

開発おける「ビルド (`build`)」と「ブートストラップ (`bootstrap`)」は、対象とする作業の「レイヤー（領域）」が異なります。

### `build`（ビルド）：アプリケーション層の準備
*   **役割**: TypeScriptからJavaScriptへのトランスパイルや、アプリで使うライブラリのインストール、Dockerイメージの作成など「アプリケーションのコードを動ける状態（成果物）にする」作業です。
*   **対象**: 自分の手元のコード自体。

### `cdk bootstrap`：AWSインフラ層の受け入れ準備
*   **役割**: ビルドされた成果物を一時的に置くためのS3バケットの作成や、AWSリソースを作る権限（IAM）の設定など、「AWS上に展開するための土台作り」です。
*   **対象**: AWS側の環境そのもの。

### 全体の流れでの位置付け
1.  **`build` (ビルド)**: コードをデプロイ可能な成果物（Asset）に変換する。
2.  **`cdk synth` (合成)**: CDKコードからCloudFormationの設計図（ Cloud Assembly / `manifest.json` ）を生成する。
3.  **`cdk deploy` (デプロイ)**: 
    *   この時、AWS側の受け入れ準備ができていなければ、先に **`cdk bootstrap`** でS3バケット等の土台を作っておく（通常は環境ごとに最初の1回のみ）。
    *   土台のS3に、ビルドされた成果物（Asset）をアップロードする。
    *   CloudFormationを通じて最終的なインフラを構築する。

**一言で言うと：**
*   **Build**: 自分のコードを「AWSに持っていける形」に変換すること。
*   **Bootstrap**: その成果物を受け取るために「AWS側に箱（S3や権限）」を準備すること。
