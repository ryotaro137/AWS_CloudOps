## 学習内容

- **Aurora Global Database / DR設計**
    - **概要**: 物理レプリケーションを用いて複数のリージョンにまたがる単一のAuroraデータベースを構築する。
    - **試験ポイント**: リージョン障害対策（DR）の決定打。ラグが1秒未満である点と、Multi-AZ（HA）との目的の違いが頻出。
    - **実務での使い方**: BCP対策としてのリージョン間DR、またはグローバル展開時の読み取り遅延解消。
    - **副業での需要**: ★★★★★。信頼性の高いインフラ設計・提案スキルとして、高単価案件の獲得に直結する。
    - **要約**: リージョン障害には「高可用性（Multi-AZ）」ではなく「DR（クロスリージョン）」が必要。Aurora Global Databaseは通常1秒未満のラグでこれを実現し、RPO要件を確実に満たす。

- **ElastiCache for Redis / クラスターモード**
    - **概要**: データをシャードに分割して複数のノードに分散配置する構成。
    - **試験ポイント**: 水平スケーリングとシャワーディング。読み取りだけでなく「書き込み」の負荷分散が必要な場合の最適解。
    - **実務での使い方**: スパイクが発生するイベント時のスループット維持。稼働中のオンラインスケーリング。
    - **副業での需要**: ★★★★☆。パフォーマンスチューニングや大規模アプリの基盤改善案件で必須の知識。
    - **要約**: クラスターモード有効 ＝ 水平スケーリング（シャード追加）が可能。急なトラフィック増でも、負荷分散によりレスポンス性能を維持できる。

- **ALB アクセスログ**
    - **概要**: ロードバランサーが処理した全リクエストの詳細（送信元IP、パス、ステータス、応答時間等）を記録する機能。
    - **試験ポイント**: CloudWatch Metricsでは分からない「個別のリクエスト元」の特定に必須。ログの保存先はS3。
    - **実務での使い方**: 5xxエラーの原因調査や、特定のIPからの攻撃特定。Amazon Athenaと組み合わせてSQLで分析するのが鉄板。
    - **副業での需要**: ★★★★★。障害調査やセキュリティ診断において、事実ベースの根拠を示すために不可欠。
    - **要約**: リクエストごとの詳細情報を記録し、トラフィック急増時の原因調査や詳細なパターン分析を可能にする。

- **Global Accelerator vs S3 Transfer Acceleration**
    - **概要**: 用途に応じた高速化サービスの使い分け。GAはNLB/ALB/EIPが対象。S3には専用のS3TAを使用する。
    - **試験ポイント**: 「S3の高速化にGlobal Acceleratorを使う」という選択肢は定番のひっかけ。S3なら Transfer Acceleration 一択。
    - **実務での使い方**: マルチリージョン展開時のレイテンシ低減（GA）と、世界中のユーザーからの大容量ファイルアップロード高速化（S3TA）。
    - **副業での需要**: ★★★★☆。適切なサービス選択によるコスト・パフォーマンス最適化提案に必須。
    - **要約**: Global AcceleratorはALB/NLB/EIP用、S3の高速化には専用のS3 Transfer Accelerationが適任。

- **RDS / 既存インスタンスのマルチAZ化**
    - **概要**: 既存のシングルAZ構成のデータベースを変更（Modify）し、マルチAZを有効化する機能。
    - **試験ポイント**: 監査指摘への迅速な対応、最小の運用管理で高可用性（自動フェイルオーバー）を実現する際の最適解。
    - **実務での使い方**: 設定変更だけでDNSエンドポイントを維持したまま可用性を向上。切り替え時のダウンタイムはフェイルオーバーに伴う短時間のみ。
    - **副業での需要**: ★★★★★。低リスク・短時間でインフラの信頼性を向上させる「即効性のある改善提案」として価値が高い。
    - **要約**: 迅速性、低コスト、可用性、最小影響のすべての観点から、既存DBのマルチAZ変換は監査対応等の急務において最も現実的で強力な選択肢となる。

- **SAML フェデレーション / LDAP統合**
    - **概要**: 既存のLDAPやActive Directory(AD)の認証情報を使い、SAML 2.0を介してAWSへのログイン（SSO）を実現する構成。
    - **試験ポイント**: 「既存のグループ構造を活用」や「オンプレミス認証情報を使用」というキーワードに対して、SAMLフェデレーションが最適解。Direct Connect（単なる専用線）やSimple AD（統合不足）との違いが頻出。
    - **実務での使い方**: AD FSやOkta等のIdPと連携し、LDAPグループをIAMロールへマッピング。社員が既存の社内アカウントでAWSコンソールへ入れるよう統制。
    - **副業での需要**: ★★★★☆。エンタープライズ企業のAWS導入・統制案件で必須。セキュリティレベル向上とユーザー管理コスト削減の両立として提案価値が高い。
    - **要約**: SAML 2.0により既存リソース（LDAP）とAWSを連携。グループマッピングにより、運用負荷を抑えつつセキュアなアクセス制御とシングルサインオンを実現する。

- **CloudWatch Agent / procstatプラグイン**
    - **概要**: CloudWatchエージェントの機能で、特定のプロセスごとのCPU、メモリ、PIDなどの詳細情報を収集する。
    - **試験ポイント**: 標準メトリクス（ハイパーバイザー測定）では不可能な「OS内部のプロセスレベル監視」を実現する際の正解。
    - **実務での使い方**: メモリリークの早期発見や、特定アプリケーションの死活監視とアラーム連携。
    - **副業での需要**: ★★★★★。障害の「予兆」を捉える高度な監視設計として、運用保守案件の価値を高める。
    - **要約**: procstatを使えば、手動運用や複雑なコードなしで、特定のプロセスの挙動を自動・継続的に可視化・監視できる。

- **AWS Cloud Development Kit (CDK)**
    - **概要**: 慣れ親しんだプログラミング言語を利用して、AWSリソースをコード（IaC）で定義・管理するフレームワーク。内部的にはCloudFormationとして実行される。
    - **試験ポイント**: 言語の選択肢（TypeScript等）や、抽象化レベル（Constructs L1-L3）、`cdk synth` 等の基本コマンドが中心。
    - **実務での使い方**: 大規模環境における構成のテンプレート化、テスト自動化、使い慣れた言語ライブラリとの統合による高度なロジック実装。
    - **副業での需要**: ★★★★★。Terraformと並び、モダンな開発案件で最も求められるIaCスキルの1つ。
    - **要約**: CDKを使えば、YAML地獄から解放され、アプリケーション開発と同じ感覚でセキュアかつ再利用性の高いインフラを高速に構築できる。

- **AWS Service Quotas / VPC制限**
    - **概要**: AWSアカウントやリージョンごとに設定されているリソースの上限（クォータ）。VPCのデフォルト上限は1リージョンあたり5個。
    - **試験ポイント**: 「既存のアプリは動いているが、新規スタックのデプロイだけ失敗する」というシナリオでは、リソース上限（Service Quotas）への到達が正解の筆頭候補。
    - **実務での使い方**: 大規模なマルチアカウント・マルチリージョン運用を開始する前に、必要なリソース（VPC、EIP、インスタンス等）の上限緩和申請を計画的に行う。
    - **副業での需要**: ★★★★☆。トラブルシューティング案件において、「設定ミス」ではなく「AWS側の制限」という盲点を指摘できる冷静な分析力は、プロとして信頼される。
    - **要約**: 同一リージョンで複数のVPCを個別に作成し続けると、デフォルト制限（5個）により新規デプロイが失敗する。既存アプリが正常な場合は、Service Quotasの確認が最優先。

- **Amazon EventBridge / SES 連携**
    - **概要**: イベント駆動でメール送信を行う際の制約。SESはEventBridgeの「直接のターゲット」にはなれない。
    - **試験ポイント**: 「EventBridgeのターゲットとしてSESを直接指定する」という選択肢は**間違い**。必ず **Lambda関数** などを中継させて、SDK経由でSESを呼び出す必要がある。
    - **実務での使い方**: 異常検知イベントを受けて、Lambdaでメール文面をリッチに加工（整形）した上でSESから通知する。
    - **副業での需要**: ★★★★☆。通知システムの共通基盤構築など、サーバーレス開発の定石として必須。
    - **要約**: EventBridgeはSESを直接叩けない。Lambdaをブリッジとして使い、SDKでメール送信ロジックを実装するのが標準的な構成。

- **AWS Control Tower vs AWS Organizations (SCP)**
    - **概要**: ガバナンスの自動化(Control Tower)と、権限の境界制限(SCP)の違い。
    - **試験ポイント**: Control Towerは「ガードレール（予防的・発見的）」を管理する上位サービス。SCPはOrganizationsの機能の一つで「予防的」制限のみを行う。Control Towerの裏側でSCPが使われている。
    - **実務での使い方**: 複数アカウントのセットアップ（Landing Zone）を自動化したいならControl Tower。特定のアクション（例：S3の公開禁止）を全アカウントで禁止したいだけならSCP。
    - **副業での需要**: ★★★★★。中〜大規模な企業のAWS統制案件でセットで問われる。安全な共通基盤を「設計・運用」できるスキルは非常に高単価。
    - **要約**: Control Towerは管理者向けのオーケストレーションツール。内部的にSCP（予防）やAWS Config（発見）を組み合わせて「ガードレール」を実現している。

- **AWS Backup**
    - **概要**: AWSサービス全体（EBS, RDS, EFS, DynamoDB, S3等）のデータ保護を中央集中型で自動化するフルマネージドサービス。
    - **試験ポイント**: 各サービス個別のスナップショットではなく「中央管理」がキーワード。**Backup Vault Lock** による「削除不能（イミュータブル）」なバックアップ（ランサムウェア対策）が頻出。
    - **実務での使い方**: タグベース（例：`Backup: Daily`）で自動的にバックアップ対象へ追加する運用。クロスリージョン・クロスアカウントコピーによるDR（災害復旧）対策。
    - **副業での需要**: ★★★★★。コンプライアンス遵守や「万が一への備え」を自動化する提案は、信頼性が全てのBtoB案件で必須。
    - **要約**: 散らばったバックアップ運用を1箇所に集約。ポリシー設定だけで、複数のAWSリソースの世代管理や改ざん防止を自動で行える。

- **S3 サーバーアクセスログ**
    - **概要**: S3バケットに対するすべてのリクエスト（GET, PUT, DELETE等）の詳細を記録する無料の機能（ストレージ料金のみ発生）。
    - **試験ポイント**: CloudTrail（API実行履歴）との違い。Trailは「誰がバケットを操作したか」を追跡するが、アクセスログは「どのオブジェクトに誰がアクセスしたか」というHTTPレベルのリクエストを記録する。
    - **実務での使い方**: 不正アクセスの調査や、特定のファイルへのアクセス頻度の分析。Amazon Athenaを使用して大量のログから特定のIPやキーを抽出する。
    - **副業での需要**: ★★★★☆。セキュリティ監査やコスト削減（「実は誰も使っていない巨大ファイル」の特定）において、事実ベースのデータを提示する際に貢献する。
    - **要約**: バケットへの全アクセス（リクエスト）を詳細に記録。ログは別のS3バケットへ保存され、セキュリティや利用動向の精密な分析に利用される。

- **EC2 インスタンス制限 / Service Quotas**
    - **概要**: 各リージョンにおけるオンデマンドインスタンスの起動数（vCPU数）にはデフォルトの上限が設定されている。
    - **試験ポイント**: 「多数のインスタンス起動に失敗する」シナリオでの原因特定。最近の試験ではインスタンス数ではなく「vCPU数」に基づく制限として問われることが多い。
    - **実務での使い方**: 大規模なキャンペーンや移行プロジェクトの前に、Service Quotasコンソールから「上限緩和申請」を行う。承認には数日かかる場合があるため、計画性が重要。
    - **副業での需要**: ★★★★★。構築当日の「起動できない」トラブル（VcpuLimitExceeded）を未然に防ぐアドバイスは、経験豊富なエンジニアとして非常に信頼される。
    - **要約**: 多数のEC2を起動する際は、デフォルトのクォータ（制限）に注意。Service Quotasからの事前申請が必要で、余裕を持ったスケジュール管理が不可欠。

- **AWS Config vs AWS CloudTrail**
    - **概要**: 「リソースの状態（Config）」と「操作の履歴（CloudTrail）」の違い。
    - **試験ポイント**: Configは「設定が規約に沿っているか（静的な状態監視）」を、CloudTrailは「誰がいつ何をしたか（動的な操作監視）」を記録する。Configのタイムライン機能で、過去の特定時点のリソース設定を確認できる点が頻出。
    - **実務での使い方**: 障害発生時、CloudTrailで「誰が変更したか」を特定し、Configで「その変更によって具体的にどう設定が変わったか（Before/After）」を確認する。
    - **副業での需要**: ★★★★★。コンプライアンス遵守の自動化（Config Rule）や、不正操作の追跡（CloudTrail）は、信頼性の高いシステム運用においてセットで必須。
    - **要約**: その通り。Configは「リソースがどう変化したか（状態）」、CloudTrailは「APIがどう呼ばれたか（操作）」を監視する。
