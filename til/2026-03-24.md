## AMIとバックアップ運用 (EC2, Lambda, EventBridge)
- **概要**: AMI（Amazon Machine Image）はEC2起動用のテンプレート。作成時の`reboot`パラメーター制御により、ファイルシステムやDBのデータの整合性を担保しつつバックアップを取得できる。
- **試験ポイント**: `CreateImage`での`reboot=true`(デフォルト)による整合性確保の仕様や、AWS Backup・DLM・EventBridge+Lambdaを用いたバックアップ自動化の実装要件や使い分けが問われやすい。
- **実務での使い方**: 本番DB稼働中で再起動ダウンタイムが許容されない場合、事前にDBロック等をかけてから`NoReboot=True`でAMIを作成し、さらに古いAMI・スナップショットの登録解除と削除も含めてLambdaで自動化（コスト保護）するのが現場のベストプラクティス。
- **副業での需要**: 需要度★★★★☆。手動運用の環境にEventBridge+Lambdaでバックアップ自動化を導入する案件や、AWS Backupへの統合（Terraform等によるIaC化）案件が豊富。古いバックアップの適正なパージ（コスト削減）まで提案できると重宝される。単価目安: 5万〜20万円。

## EFS (Elastic File System) とマウントトラブルシューティング
- **概要**: 複数EC2間で共有可能なマネージドNFS（ポート2049）。別AZのEC2からマウントするには、各AZごとのマウントターゲットと、適切なインバウンド/アウトバウンドを許可するセキュリティグループが必要。
- **試験ポイント**: 「新しいAZからEFSがマウントできない」場合のトラブルシューティングが頻出。原因の大半は「マウントターゲットの作成漏れ」か「SGで2049ポートの受信が許可されていないこと」のいずれか。
- **実務での使い方**: ロードバランサ配下の複数Webサーバー（WordPressなど）の静的アセット共有ストレージとして利用。AZを拡張する際にマウントターゲットやSGの作成漏れを防ぐため、Terraform等のIaCでセット構築のモジュール化を行うのがベストプラクティス。
- **副業での需要**: 需要度★★★☆☆。自作NFSサーバーからマネージドなEFSへの移行案件や、TerraformによるEFS構築・整理案件がある。データ移行（DataSync等）とセットで提案できると好印象。

## AWSタグの管理とコスト配分タグ構成
- **概要**: AWSリソースに付与するキーと値のペア（最大50個）。ユーザー定義タグ（部門や環境など）とAWS生成タグがある。タグエディタで複数リソースに一括付与・管理が可能。
- **試験ポイント**: 「部門ごとのコストを可視化したい」という要件に対し、「ユーザー定義タグ（部門名）を付与」→「タグエディタで既存リソースに一括付与」→「Billingコンソールでコスト配分タグとして有効化」という一連のプロセスが超頻出パターン。
- **実務での使い方**: 「Environment(環境)」「Project」「Owner」等のタグ付けを全社ルール化し、付与されていないリソースの作成をSCP（Service Control Policy）で強制ブロックさせたり、AWS Configで非準拠リソースとして検知・修正する運用が、大企業のベストプラクティス。
- **副業での需要**: 需要度★★★★☆。複数システムが相乗りし「コストのブラックボックス化」に悩む長期運用アカウント向けに、タグ設計ガイドラインの策定からタグエディタでの一括付与、Cost Explorerでの部門別ダッシュボード化までを一気通貫で支援するコスト最適化案件が需要大。

## Amazon ElastiCache for Redis の設計と可用性
- **概要**: 高速アクセス可能なインメモリデータストアのフルマネージドサービス。Memcachedと異なり、複雑なデータ型のサポート、データ永続化（自動バックアップ等）、高可用性機能（マルチAZでの自動フェイルオーバー等）を備える。
- **試験ポイント**: 「高可用性とフォールトトレランスを実現する構成」を問う問題が超頻出。正解は「マルチAZ配置（自動フェイルオーバー）」。また「クラスターモード＝スケーラビリティ（可用性ではない）」「Auto Scalingは適用不可（マネージドサービスのため）」の引っかけにも要注意。
- **実務での使い方**: RDSなどの背後にあるメインDBの「手前」に配置し、読み取りクエリやセッションキャッシュを高速化・負荷分散するために用いる。本番環境ではマルチAZ構成による自動フェイルオーバーと、日次のスナップショットによるデータ保護を必ず有効化する。
- **副業での需要**: 需要度★★★☆☆。EC2やRDSの単純構成で起きたDBのパフォーマンス限界（スロークエリやスパイク等）に対し、ElastiCache (通常はRedis) を導入して負荷分散を図る改修や、既存MemcachedからRedisへの移行案件などが存在する。

## Amazon ElastiCache for Memcached とスケーリング戦略
- **概要**: シンプルなキー・バリューストアで構成される揮発性インメモリストレージ。Redisと違い、永続化機能や複雑なデータ型、自動フェイルオーバーを持たない。
- **試験ポイント**: CloudWatchの `Evictions` メトリクス（メモリ不足による古いデータの強制削除回数）の悪化に対する解決策が頻出。正解は、「ノードのインスタンスタイプを大きくする（スケールアップ）」か「クラスターにノードを追加する（スケールアウト）」。
- **実務での使い方**: クライアント側でのハッシュアルゴリズム（データ分散）実装が必要。一時的なセッション情報の管理や、消えても再取得可能なマスタデータのキャッシュ等、シンプルな設計に適している。`Evictions` や `CacheMisses` を監視し、不要なデータを適正にパージするTTL（有効期限）の最適化が必須。
- **副業での需要**: 需要度★★☆☆☆。機能の制約から新規要件ではRedisが採用されることが多いため、Memcached自体の構築案件は少なめ。古いシステムの保守運用や、既存の「DB高負荷改善」のスポット対応（とりあえずキャッシュ層を挟んで急場を凌ぐ要件）で時折遭遇する程度。

## Security Hubと一元的なセキュリティ管理 (CIS Benchmark, Organizations)
- **概要**: 複数AWSアカウントのセキュリティスコアやアラート（GuardDuty等からの検出結果）を統合し、「CIS AWS Foundations Benchmark」などの標準コンプライアンス要件に準拠しているかを自動評価するサービス。
- **試験ポイント**: 「複数アカウントのセキュリティ評価（CIS等）を一元管理したい」という要件に対し、「Security HubとAWS Organizationsを統合する」構成が頻出。他のセキュリティ系サービス（脅威検知=GuardDuty、脆弱性スキャン=Inspector、構成変更管理=Config）との用途の違いもしばしば問われる。
- **実務での使い方**: 組織の一元管理としてOrganizations側でセキュリティ管理の専用アカウント（Delegated Administrator）を登録し、子アカウント側でのSecurity Hubの有効化を自動化（自動メンバー登録）する。その後CIS Benchmarkで検出されるNG設定（MFA未設定、不用意なアクセスキー、全開きのSGなど）に対し修復（Remediation）ワークフローを回すのが定番。
- **副業での需要**: 需要度★★★★★。企業がAWS利用を拡大したり監査対応を強化する際、コンプライアンス可視化は急務となるため、Security Hubの全社導入（Organizations連携）や、CIS Benchmarkに従った既存環境のアセスメントおよび改修（IaCコードの修正や設定変更）案件は単価も非常に高く（20万円〜/件）、継続支援にも直結しやすい。

## Amazon RDSの自動バックアップとリードレプリカ昇格
- **概要**: RDSのデータ保護には、自動バックアップ（日次スナップショット＋トランザクションログ）を用いた「任意の時点への復元（ポイントインタイムリカバリ）」と、非同期で同期されている「リードレプリカを独立したDBへ昇格（プロモート）させる」という2つのアプローチがある。
- **試験ポイント**: 要件によって正解が分岐する頻出問題。「運用負荷を最小限にしつつ、過去24時間以内の**ある特定の時点**のデータに復元したい」なら自動バックアップからの復元。「運用のオーバーヘッドを最小限にし、**最速（数分以内）で**復元（切り替え）したい」ならリードレプリカの昇格が正解。（※pg_dump等の手動運用は運用負荷が高いため基本的に不正解）
- **実務での使い方**: 本番環境では自動バックアップ（保持期間7〜35日設定）の有効化が必須。リードレプリカは通常「読み取り負荷の分散」として使われるが、プライマリDBが致命的な論理障害・物理障害を起こした緊急時のDR（災害復旧）対策として、リードレプリカを即座に「昇格」させ、新プライマリとしてアプリからの接続エンドポイントを切り替える運用手順を用意しておくのがベストプラクティス。
- **副業での需要**: 需要度★★★★★。RDSのバックアップ・DR（災害復旧）戦略はあらゆるシステムで必須要件。顧客のRPO（目標復旧時点）とRTO（目標復旧時間）をヒアリングし、「自動バックアップ復旧」「リードレプリカ昇格」「マルチAZ」を組み合わせて最適なアーキテクチャを提案・Terraform実装するスキルは非常に市場価値が高く、高単価案件（数十万〜）の上流工程に直結する。
