## 学習内容

## ACM (AWS Certificate Manager) の監視機能
- **概要**: ACMで管理する証明書の有効期限切れ（失効）を防ぐため、CloudWatchメトリクス（`DaysToExpiry`）とEventBridge、SNSを連携して管理者に自動通知する監視・アラートの仕組みです。
- **試験ポイント**: CloudWatchメトリクスの`DaysToExpiry`を用いた証明書の有効期限監視フロー（ACM → EventBridge → SNS）の自動化構成や、運用自動化による信頼性向上の設計がよく問われます。
- **実務での使い方**: EventBridgeを用いて「`DaysToExpiry` < 14」などの特定条件でイベントを検知し、SNSトピック経由で管理者のメールアドレスにプッシュ通知を自動送信するイベント駆動型アーキテクチャを構築します。証明書失効によるシステム停止などのクリティカルトラブルを未然に防ぎます。
- **副業での需要**: ★★★★★ (監視・アラート設計)。システム運用においてSSL/TLS証明書の更新管理は必須課題であり、CloudWatchやEventBridgeを活用した自動監視・通知（手動運用からの脱却）の導入・改善案件は非常に需要が高いです。

## Amazon S3 の暗号化オプション
- **概要**: S3におけるデータ保護の仕組み。AWSがキーを完全管理する「SSE-S3」、KMS連携で監査に強い「SSE-KMS」、顧客がキーを提供する「SSE-C」、およびクライアント側で暗号化する方式の4種類があります。
- **試験ポイント**: 要件に応じたオプションの使い分けが頻出します。「CloudTrailで詳細な監査証跡（キーの使用履歴）を取りたい」場合は **SSE-KMS** が正解になり、「顧客自身が暗号化キーを毎リクエストで提供・管理したい」場合は **SSE-C** や **クライアント側暗号化** が問われます。
- **実務での使い方**: 基本的かつシンプルな設定として、無料の **SSE-S3** をデフォルト暗号化として適用します。金融機関や医療系システムなど、厳格なコンプライアンス要件があり、キーのローテーションや詳細なアクセス制御・監査ログが必要な環境では **SSE-KMS**（カスタマーマネージドキー）を採用します。
- **副業での需要**: ★★★★ (セキュリティ運用)。S3バケットのデフォルト暗号化（SSE-KMS等）の設定は、インフラ構築・IaC化案件におけるセキュリティのベストプラクティスとして必ず求められます。コンプライアンス対応としての単価アップにも繋がります。

## Auto Scaling の種類と特徴
- **概要**: EC2の負荷に応じてインスタンス数を自動増減させる仕組み。リアルタイム監視に基づく「動的スケーリング」、時間指定の「スケジュールベース」、機械学習で需要予測を行う「予測スケーリング」の3種があります。
- **試験ポイント**: 「CPU使用率50%超過時に即座に対応したい」といった突発的・リアルタイムな要件には**動的スケーリング（特にターゲット追跡スケーリング）**が正解になります。スケジュールベースや予測スケーリングは計画的な負荷変動には有効ですが、即時対応には適していません。
- **実務での使い方**: 基本的かつ推奨される設定として、シンプルで効果の高い「ターゲット追跡スケーリング」を使用し、目標値（例: CPU使用率50%）を維持するように構成します。安定稼働のためには、高速起動可能なAMIの作成、スケールイン保護、適切なクールダウンとヘルスチェック猶予期間の設定が重要です。
- **副業での需要**: ★★★★★ (IaC / 監視・アラート設計)。可用性確保とコスト最適化の両立を目的としたAuto Scalingの手動構築からのIaC化（CloudFormation/Terraform）案件は非常に需要が高いです。安全にスケーリングできる環境を構築・テストするスキルが高く評価されます。

## AWS Global Accelerator の概要
- **概要**: Global Accelerator は、AWS のグローバルネットワークインフラを活用し、2つの Anycast 静的 IP アドレスを提供して、ユーザーからのトラフィックを最適な AWS リージョンへ高速にルーティングします。DNS ルックアップ遅延を排除し、接続レイテンシを低減します。
- **試験ポイント**: Global Accelerator の 2 つの静的 IP、エンドポイントとして ALB/NLB を設定できる点、トラフィックの最適ルーティングとヘルスチェックの仕組みが試験で問われます。
- **実務での使い方**: ALB や NLB をエンドポイントに設定し、固定 IP を介してグローバルに高可用性・フォルトトレラントなアプリケーション配信を実現します。マルチリージョン構成での DR や、ユーザー体験向上のためのレイテンシ最適化に利用。
- **副業での需要**: ★★★★★ (ネットワーク最適化・高可用性設計)。グローバルに展開する SaaS やゲーム、金融系サービスでの低遅延・高可用性要件に対し、Global Accelerator の導入・設定案件が高単価で需要があります。

## CloudWatch Logs のデータ保護ポリシー
- **概要**: ログデータ内の機密情報（PII：個人情報や PHI：医療情報など）を自動検出し、マスキング（保護）したり、検出時にアラートや特定のアクションを起動させることができる機能です。
- **試験ポイント**: 「ログ内の機密情報を保護したい」「機密データがログに書き込まれた際に検知・対応したい」という要件の最適解として問われます。Macie は S3 のデータ保護ですが、CloudWatch Logs 内のデータであれば本機能が正解です。
- **実務での使い方**: アプリケーション側でのマスキング漏れに備え、フェイルセーフとして導入します。クレジットカード番号や認証情報の流出リスクを抑え、検出時には EventBridge や SNS を経由して即座に管理者にアラート通知する構成がベストプラクティスです。
- **副業での需要**: ★★★★ (セキュリティ運用)。ログ基盤の構築や運用案件において、GDPR、PCI DSS などのコンプライアンス要件が厳しいシステム（金融、ヘルスケア、EC サイトなど）で提案・導入するケースが多く、セキュリティ担保としての価値が高いです。

## VPC BPA (Block Public Access)
- **概要**: 最小限の運用負荷で、VPCリソースに対するパブリックインターネットアクセスを防ぐ強固なセキュリティ機能です。Ingress-onlyモードでは、VPCからのアウトバウンドトラフィックは許可しつつ、インターネットからのインバウンドトラフィックをすべてブロックします。
- **試験ポイント**: 「運用上のオーバーヘッドを最小限に抑えつつ、VPCへのインバウンドアクセスをすべてブロックしたい」場合の最適解として問われます。他のいかなる設定（セキュリティグループやNACL等）よりも優先される点が重要です。
- **実務での使い方**: 新規AWSアカウントの初期設定時や、社内専用のシステム（閉域網要件）など、意図しない外部公開やパブリックアクセスの設定ミス（ヒューマンエラー）を防ぐための強力なガードレールとして組織全体に適用します。
- **副業での需要**: ★★★★ (セキュリティ運用)。S3のBPAと同様に、インフラ構築時のセキュリティのベストプラクティス・基本設定として求められます。AWS Organizationsを使ったマルチアカウント展開時の自動設定要件にもよく含まれます。

## ALB と NLB の使い分け (レイヤー7 vs レイヤー4)
- **概要**: ロードバランサーの特性の違い。ALB（Application Load Balancer）はL7でHTTP/HTTPSパスベースのルーティングやヘルスチェックが行え「ウェブ階層」に最適です。一方、NLB（Network Load Balancer）はL4(TCP/UDP)で動作し、数百万リクエスト規模の超高性能処理が可能で「API階層」に最適です。
- **試験ポイント**: 要件に応じた使い分けが頻出します。「HTTPヘルスチェック」「パスやホストに対する柔軟なルーティング」があれば ALB が正解、「数百万規模の超高負荷トラフィック」「超低遅延」「TCP/UDP ヘルスチェック」といった要件であれば NLB が正解です。
- **実務での使い方**: Webバックエンド（フロントエンドからのリクエスト）には ALB を配置してパスベースの柔軟な振り分けを行い、マイクロサービス間通信やリアルタイム性の高いゲームサーバー、大規模 API 層には NLB を配置するなど、多層アーキテクチャ（階層ごとの適材適所）で併用します。
- **副業での需要**: ★★★★★ (IaC / ネットワーク構築)。可用性の高いWebシステム構築において ALB はほぼ必須のコンポーネントであり、また高負荷な API サーバー基盤の構築やパフォーマンス改善案件などでは NLB の設計スキルの需要が高くなります。

## CloudFormation StackSets のドリフト検出
- **概要**: StackSets 内のスタックインスタンス（複数アカウントやリージョンに展開されたリソース）が、元の CloudFormation テンプレートの設定から意図せず変更（ドリフト）されていないかを検出する機能です。
- **試験ポイント**: 「運用上のオーバーヘッドを最小限に抑えつつ、複数アカウント・リージョンにまたがるリソースの設定変更を特定したい」要件の最適解として問われます。"通常の更新プロセス以外での変更（手動操作などによるドリフト）" を検出できる点がキーワードです。
- **実務での使い方**: AWS Organizations 環境下でのガバナンス維持に使用します。マネジメントコンソール等で直接インフラが変更された箇所を洗い出し、IaCの整合性を監査・修復する運用管理フローに組み込みます。
- **副業での需要**: ★★★★★ (IaC / 監視・アラート設計)。CloudFormation (StackSets) や Terraform などを活用したインフラ自動化・マルチアカウント構築において、コードと実環境の差分（ドリフト）を検知・管理する仕組みはエンタープライズ案件を中心に高い需要があります。
