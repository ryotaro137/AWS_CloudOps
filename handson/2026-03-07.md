## SOA（SysOps Administrator）ハンズオンで高額課金に注意すべきサービスと回避方法

SOAの試験範囲である運用やネットワーク、セキュリティのハンズオンでは、起動時間単位で高額な料金が発生するサービスや、構成によって請求が跳ね上がるサービスが多く登場します。これらを使用する際は**「必要なときだけ構築し、終わったら即座に削除する」**のが鉄則です。

### 1. NAT Gateway (ネットワーク系)
- **コストの罠**: 起動しているだけで料金が発生します（東京リージョンで約 $0.062/時）。さらにデータ処理料金もかかります。1ヶ月放置すると約5,000〜6,000円。
- **回避・節約方法**:
    - ハンズオンが完了したら**最優先で削除**する。
    - 単なるAWSのAPI呼び出しであれば、NAT Gatewayの代わりに「VPCエンドポイント」を使用できないか検討する。

### 2. Amazon RDS / Aurora (データベース系)
- **コストの罠**: `Multi-AZ`（マルチAZ配置）や `Provisioned IOPS`（プロビジョンドIOPS）を選択すると非常に高額になります。またAuroraは停止状態でもストレージ料金がかかり、一定期間で自動起動します。
- **回避・節約方法**:
    - ハンズオン時は必ず **`Single-AZ`** を選択し、インスタンスサイズは `db.t3.micro` などの最小クラスを選ぶ。
    - ストレージは「汎用SSD (gp2/gp3)」を選択し、容量は20GB（最小）にする。
    - 終了後はスナップショットを取らずに「完全に削除」する。

### 3. Amazon EKS (コンテナ系)
- **コストの罠**: コントロールプレーン（クラスター自体）の管理費用として、1クラスターあたり **$0.10/時間** かかります。約月額 $70（1万円以上）。これにワーカーノード（EC2）の料金が加わります。
- **回避・節約方法**:
    - EKSの構築検証（eksctl等）が完了し、動作確認が済んだら、直ちにクラスターおよびノードグループを削除する。

### 4. AWS Directory Service (認証系)
- **コストの罠**: Active Directoryのマネージドサービスですが、「AWS Managed Microsoft AD」を選択すると、Standardエディションでも約 $0.16/時（月額 1万数千円）かかります。
- **回避・節約方法**:
    - どうしてもADのハンズオンが必要な場合は「Simple AD」の Small サイズを選ぶ（比較的安い）。
    - 検証が終わったらすぐにディレクトリを削除する。

### 5. AWS Client VPN Endpoints / Route 53 Resolver Endpoints (ネットワーク系)
- **コストの罠**: エンドポイント自体に1時間単位の利用料金（東京で約 $0.10〜0.15/時 ＋ サブネットごとの関連付け費用）がかかります。
- **回避・節約方法**:
    - 構築と通信テストが完了次第、エンドポイントを直ちに削除する。

### 6. AWS KMS (セキュリティ系)
- **コストの罠**: 「カスタマー管理キー（CMK）」を作成すると、1キーあたり **月額 $1** かかります（削除しない限り毎月）。非対称キーの暗号化・復号操作もリクエスト単位で課金されます。
- **回避・節約方法**:
    - 学習用のキーを作成したら、使い終わった段階で必ず「キーの削除をスケジュール」する（待機期間中は課金されない）。
    - 課金されたくない場合は、極力「AWS管理キー (aws/s3 など 月額無料)」をハンズオンで利用する。

### 7. AWS Config / CloudTrail (監査系)
- **コストの罠**:
    - **Config**: 記録する「設定項目の変更数」で課金されます。すべてのおよびグローバルリソースを対象にすると、意図せず変更履歴が爆発して課金されることがあります。
    - **CloudTrail**: デフォルトの管理イベントのみを1つの証跡で記録するのは無料ですが、「データイベント（S3内オブジェクトの全操作など）」を有効にリスト指定すると、リクエスト量に応じて高額な課金が発生します。
- **回避・節約方法**:
    - ハンズオン用の特定のリソース（例えば特定のタグがついたEC2だけ）にスコープを絞って記録する。
    - S3やLambdaの「データイベント」の記録は有効にしない（特に自身で負荷テストなどを行う場合は注意）。

## Infrastructure trouble-shooting on AWS Vol.1 (VPC/EC2トラブルシューティング)
- **概要**: 掲示板サービス（WebおよびDBサーバー）間の通信障害など、EC2とVPCを中心としたネットワークトラブルシューティング。高額なNAT Gatewayを使用せず、SSM Session Manager等を利用してコストを極小化する学習アプローチ。
- **試験ポイント**:
    - **通信要件のトラブル解決**: 「WebからDBに繋がらない」というシナリオは最頻出。選択肢の中から「セキュリティグループのインバウンド設定」「ネットワークACLのアウトバウンド/インバウンド設定（ステートレスであることの理解）」「ルートテーブルのターゲット設定」のどれが根本原因かを切り分ける能力が問われる。
    - **Reachability Analyzer**: 通信経路のトラブルシューティングツールとして「VPC Reachability Analyzer」を使う選択肢が正解になるパターンがよく出題される。
    - **SSM経由でのEC2アクセス**: SSHキーを使わずにEC2へログインできないトラブルでは、「Session Managerに必要なIAMポリシー (`AmazonSSMManagedInstanceCore`) がアタッチされていない」または「VPCからSSMエンドポイント（またはインターネット）への経路がない」ことが正解になる。
- **実務での使い方**: 
    - **安全なネットワーク設計**: 実務では「とりあえず0.0.0.0/0で全許可する」のは絶対NG。WebのSGからはインターネット(0.0.0.0/0)の443を許可し、DBのSGは「WebのSGからのみ」許可するという、リソース間の参照によるアクセス制御がベストプラクティス。
    - **フローログでの原因特定**: トラブル時は闇雲に設定を変えるのではなく、VPCフローログをCloudWatchでクエリし、「REJECT」されている通信のIPとポートを確認してからSG/NACLを修正するアプローチを取る。
    - **最もコスパの良いハンズオン構成（NATなし）**: 高額なNAT Gatewayを構築せず、パブリックにWebサーバー、プライベートにDBサーバーを設置し、意図的にSGやNACLの一部を間違え（Web→DBへの3306を拒否するなど）、後から原因を特定して修正する訓練を行う。終了後はEC2を即座に破棄する（費用は数十円程度）。
- **副業での需要**: ★★★★★ (ネットワーク設計・トラブル対応)。「システムが突然通信できなくなったので調査してほしい」「新しくサーバーを立てたが社内からアクセスできない」といったスポット案件は常に需要があり、短時間で解決できれば数万円〜10万円以上の単価になることもある。「推測ではなくVPCフローログやReachability Analyzerを用いて論理的に原因を特定できる」ことが強力なアピールポイントになる。

## 副業需要に直結するトラブルシューティング＆自動化ハンズオン

副業市場で非常に需要が高い**「監視・通知（CloudWatch/SNS）」「セキュリティ（AWS Config）」「コスト自動化（EventBridge）」**の3つの分野から、現場で頻発するリアルなトラブルを安全かつ低コストに再現・解決する検証シナリオ。

### シナリオ1：監視・アラート設計（副業需要 ★★★★★）
- **現場のトラブル**: 「サーバー（EC2）の負荷が急増してWebサイトが落ちていたのに、ユーザーからクレームが来るまで誰も気づかなかった。」
- **ハンズオン手法（コストほぼゼロ）**:
    1. パブリックサブネットに `t2.micro` を1台立てる。
    2. Amazon SNSでトピックを作成し、自分のメールアドレスを登録して承認する。
    3. CloudWatchで、EC2の「CPU使用率 (CPUUtilization)」が「1分間で 80% を超えたら」SNSへ通知するアラームを作成する。
    4. Session ManagerでEC2にログインし、`stress` ツールをインストールしてCPU使用率を100%にする負荷テストを実行する。
    5. アラーム状態になりメールが届くことを確認後、リソースを削除する。

### シナリオ2：セキュリティの自動監査（副業需要 ★★★★）
- **現場のトラブル**: 「開発メンバーが検証のために、データベースやEC2のセキュリティグループを一時的に `0.0.0.0/0` (全公開) に空けてしまい、そのまま戻し忘れて放置されている。」
- **ハンズオン手法（コスト数十円）**:
    1. 対象リージョンでAWS Configの設定記録をオンにする。
    2. AWS Configの「ルール」から、AWSマネージドルールである `vpc-sg-open-only-to-authorized-ports` または `restricted-ssh` （SSHのフルオープンを禁止するルール）を追加する。
    3. 新しいセキュリティグループを作成し、意図的にインバウンドで「SSH（22番）」を `0.0.0.0/0` に許可する。
    4. ConfigダッシュボードでそのSGが **「非準拠 (Noncompliant)」** として警告されるのを確認する。
    5. 確認後、ルールとSGを削除し、Configの記録設定をオフにする。

### シナリオ3：コストの自動最適化（副業需要 ★★★★）
- **現場のトラブル**: 「開発・検証環境のEC2をエンジニアが土日も夜間もつけっぱなしにしており、使っていない時間のサーバー代が無駄にかかっている。」
- **ハンズオン手法（コストほぼゼロ）**:
    1. 検証用EC2（`t2.micro`）を作成し、タグに `Env` = `Dev` を付ける。
    2. Amazon EventBridge Scheduler にアクセスし、実行タイミングをcronベース（例: 現在時刻の5分後）に設定する。
    3. ターゲットを「Amazon EC2」の「StopInstances（停止）」にし、先ほどのEC2インスタンスIDを指定する。
    4. 設定時刻になったら自動的にEC2が停止状態になるかを確認する。
    5. テスト完了後、スケジュールとEC2を削除する。
