# CloudOps 発展編：高難易度・実践シナリオ3選

これまでの学習内容（`CloudOps_Handson_Priority.md` の基礎トラブルシューティングや、`til` での高度なネットワーク・セキュリティ知識、`2026-03-08.md` のバックアップ戦略）を統合した、より実務的で難易度の高いシナリオです。

## 難易度：高（★★★★★）SRE・SecOpsの統合トラブル対応

### シナリオ5：ハイブリッドNW環境下でのセキュア基盤トラブルシューティング
- **概要**: オンプレミスからAWS上のシステム（EC2 + RDS）へVPN経由でアクセスしている環境をモダナイゼーションした直後に、複数の障害が発生した。
- **発生している問題**:
  1. 社外の提携先から「固定IPをFW（ホワイトリスト）に入れたのにAPIに繋がらない」とクレーム（ALBのIP変動問題）。
  2. セキュリティ監査で「RDSの通信が暗号化されていない」「GitHubにDBのパスワードが漏洩した痕跡がある」と指摘された。
  3. VPN利用社員から「AWS導入後、通常のWeb会議まで重くなった」と苦情が入っている。
- **解決へのアプローチ（主なハンズオン要素）**:
  - **NW改善**: AWS Client VPNの「スプリットトンネル」を有効化し、不要な通信をAWSから切り離す。
  - **アクセス改善**: AWS Global AcceleratorをALBの前段に配置してAPIへの静的（固定）エニーキャストIPを提供する。
  - **セキュリティ改修**: AWS Secrets Managerを利用してRDSの認証情報を安全に管理・動的取得し、RDSカスタムパラメーターグループで `rds.force_ssl=1` を設定して通信を暗号化する。
- **選定理由**: ネットワークの最適化（VPN）、フロントの静的IP化（GA）、DBのセキュリティ強化（KMS/Secrets Manager）という、Tilで学習した複数の領域が絡み合う、実務で非常に多い「移行後トラブル」の集大成です。
- **ハンズオン費用目安**: $1.0 〜 $2.5（約150円〜350円 / 2〜3時間程度の検証を想定）
  - *内訳・注意点*: Client VPN Endpoint（$0.10/時）、Global Accelerator（$0.025/時）、ALB（$0.0225/時）などの「起動しているだけでかかる固定費」が複数発生します。検証が終わったら速やかにリソース（特にVPNエンドポイントとGA）を削除してください。

### シナリオ6：マルチアカウント運用での高度なDR（ディザスタリカバリ）と自動監視パイプライン
- **概要**: ランサムウェア対策として、本番環境のデータを完全に分離された「監査・バックアップ専用別アカウント」にレプリケーションする機構を構築。同時に、運用チームの負担を減らすため監視アラートの自動化を図る。
- **発生している問題**:
  1. 異なるAWSアカウントにS3オブジェクトをレプリケートしたが、「宛先アカウント側でファイルが読み込めない（所有権エラー）」状態に陥っている。
  2. EKS/コンテナ環境にした後、CloudWatchアラームが鳴りすぎているが、一時的なスパイクばかりで運用チームが疲弊・形骸化している。
  3. 障害発生時のチケット手動作成が漏れる事件が発生した。
- **解決へのアプローチ（主なハンズオン要素）**:
  - **DR構築**: S3クロスアカウントレプリケーションを設定し、`Change object ownership to destination bucket owner (Replica Owner Override)` を有効化して権限問題を解決する。
  - **監視最適化**: CloudWatch異常検出（Anomaly Detection）やAmazon DevOps Guruを導入し、機械学習ベースの動的なしきい値で誤検知を減らす。
  - **通知自動化**: CloudWatch Alarm -> Amazon SNS -> (Lambda) -> チケット管理ツール（Webhook）へと繋ぐFanout構成を構築し、障害チケット起票を完全自動化する。
- **選定理由**: 複数のアカウントを跨ぐIAMの権限トラブルシューティング（S3所有権）や、スマートな運用自動化（SNSのWebhook、MLベースの異常検知）といった、CloudOpsとしての「一段上の運用設計・インシデント回避」が問われるシナリオです。
- **ハンズオン費用目安**: 数十円 〜 $1.0（約10円〜150円 / 2〜3時間程度の検証を想定）
  - *内訳・注意点*: S3の操作やSNSは無料枠内でほぼ収まります。CloudWatch異常検出（$0.30/メトリクス・月額）やDevOps Guru（$0.0042/リソース時）も数時間の検証であればごく少額です。ただし、EKSクラスター（$0.10/時）を本格的に構築すると料金が跳ね上がるため、EC2などで代用してCloudWatchアラームを発生させるのがコストを抑えるコツです。

### シナリオ7：モダンフロントエンド配信におけるキャッシュ制御と動的ルーティングの罠
- **概要**: グローバル展開するWebサービスにおいて、Amazon CloudFrontを構築したが、ユーザーからの表示崩れや古いデータ表示などに関するクレームが相次いでいる。
- **発生している問題**:
  1. 「スマホからアクセスしているのに、PC用の巨大なレイアウト画像が表示される」という問い合わせが多発。
  2. アプリのCSSやJSファイルを更新してもユーザー環境では古いデザインが表示され続け、頻繁に高コストなキャッシュ削除（Invalidation）を実行してしまっている。
  3. 特定のAPIエンドポイントで、常に古いデータ（キャッシュ）が返ってきて機能不全に陥っている。
- **解決へのアプローチ（主なハンズオン要素）**:
  - **ヘッダーベースキャッシュ修正**: CloudFrontのBehavior設定で、`CloudFront-Is-Mobile-Viewer` ヘッダー等をオリジンへ転送・キャッシュキーに追加し、デバイスごとにキャッシュを分割する。
  - **キャッシュ戦略の見直し**: 静的ファイルに対して、URLの中にバージョン番号を持たせる機構（例: `/v2/style.css`）を取り入れ、Invalidationの回数を激減させる。
  - **Cache-Control制御**: APIのレスポンスヘッダー（オリジン側）で適切に `Cache-Control: no-store` 等を設定し、CloudFront側が誤って動的データをキャッシュしないように制御する。
- **選定理由**: CDNのキャッシュ仕様とHTTPヘッダーの理解は、Webアプリケーション運用において必須のインフラ知識です。ユーザー体験（レイテンシ表示速度）とAWS転送料金（コスト）に直結するため、トラブル解決能力が非常に評価される領域です。
- **ハンズオン費用目安**: ほぼ無料 〜 数十円（約0円〜50円 / 2〜3時間程度の検証を想定）
  - *内訳・注意点*: CloudFrontは毎月1TBまでのデータ転送無料枠があり、キャッシュ削除（Invalidation）も月間1000パスまで無料です。そのため、裏側で動かすALBやEC2の稼働時間分の安価な料金（数円/時）のみで安全に検証が可能です。
