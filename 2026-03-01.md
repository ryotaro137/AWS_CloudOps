# ハンズオン記録

## 日付
2026年03月01日

## 学習内容
- CPU の負荷テストは `stress` が主流
- Amazon Linux 2023 では `sudo dnf install stress -y` でインストール可能
- `vmstat 2` でスワップや CPU 負荷（us, sy 等）の状況を把握できる
- コスパの良いコア数確認: `grep "cpu cores" /proc/cpuinfo | sort -u` や `nproc`

### 実行コマンドメモ
```bash
# インストール
sudo dnf update -y
sudo dnf install stress -y

# 1. CPU 負荷テスト (-c または --cpu)
# 2コアに対して 60秒間 100% の負荷をかける
stress -c 2 -t 60s

# 2. メモリ負荷テスト (--vm)
# 2つのワーカーでメモリを確保・解放。デフォルトは 256MB/worker
# --vm-bytes で容量指定、--vm-hang で確保状態を維持
stress --vm 2 --vm-bytes 512M --vm-hang 60s -t 60s

# 3. ディスク負荷テスト (--hdd)
# 書き込みと削除を繰り返す。デフォルトは 1GB/worker
stress --hdd 2 --hdd-bytes 1G -t 60s
```

### 負荷状況の確認・モニタリング
- **CPU**: `vmstat 2`（`id` カラムが 0 に近づき、`us` や `sy` が上がれば負荷がかかっている）
- **メモリ**: `dstat -m`（`used` が増加することを確認）
- **ディスク**: `dstat -r` または `dstat --io`（`writ` の値が増加することを確認）

### 知っておくと便利なオプション
- `-t, --timeout [秒]`: 指定した時間だけ負荷をかける。
- `-n, --dry-run`: 実際に負荷をかけず、設定が正しいか確認。
- `-v, --verbose`: 詳細なデバッグ情報を表示。

> **参考記事**: [負荷テストのための stress](https://zenn.dev/4kzknt/articles/df359a81c81789)

### 過去の CPU 使用率を調査する方法
「いつの間にか CPU が跳ね上がっていた」という過去の事象を追うためのツール群。

#### 1. OS レベル：`sar` コマンド
Linux 標準の統計記録ツール。過去に遡って CPU、メモリ、I/O 等の状況を確認できる。

```bash
# インストールと有効化 (Amazon Linux 2023)
sudo dnf install sysstat -y
sudo systemctl enable --now sysstat

# 本日の過去の CPU 推移を表示
sar -u

# 特定の日 (例: 28日) の推移を表示 (ログファイル /var/log/sa/saXX を指定)
sar -u -f /var/log/sa/sa28

# 1秒おきに 5回、現在の CPU 使用率を表示
sar 1 5
```

#### 2. AWS レベル：CloudWatch Metrics
インスタンスの外側から監視しているため、OS がハングアップしている最中でもデータが確認できるのが強み。

- **確認方法**: EC2 マネジメントコンソールの「モニタリング」タブ、または CloudWatch コンソールから `CPUUtilization` メトリクスを確認。
- **詳細モニタリング**: デフォルトは 5 分間隔。より細かいスパイク（1 分間隔）を追いたい場合は「詳細モニタリング」を有効化する。

> **CloudOps実践テクニック**
> 1. まず **CloudWatch** で「いつ発生したか」の時刻を特定する。
> 2. その時刻を元に、OS 内の **`sar`** や各種ログ（`/var/log/messages` など）を確認し、暴走プロセスの特定や不審なアクセスの有無を突き止める、という「俯瞰から詳細へ」の切り分けが定石。
